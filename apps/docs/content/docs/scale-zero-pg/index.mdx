**scale-zero-pg** is a scale-to-zero PostgreSQL platform for Kubernetes. An idle
database costs **zero compute**; a client TCP connection wakes it in **~2.5s** (or
**~400ms** from an opt-in warm tier). It is the **database layer** of the knext
platform — knext scales your *app* to zero, scale-zero-pg scales its *database* to
zero, and the two wake together on a single visitor request.

## How it works

Native PostgreSQL compute on [Neon's open-source](https://github.com/neondatabase/neon)
disaggregated storage, all on Kubernetes — with one small piece of glue we build:

- **Storage plane** (never scales to zero): safekeepers hold a quorum-replicated WAL,
  a pageserver serves pages on demand with S3/object-storage offload. This makes the
  Postgres compute *stateless*.
- **Compute plane**: one stateless Neon compute per database, `replicas: 0` at rest,
  scaled 0↔1 on demand. Cold start = attach + lazy page fetch, independent of database
  size — no restore.
- **Routing plane** (`gateway/`, Go): a wake-on-connect Postgres proxy. It parses the
  startup packet, wakes the compute through the Kubernetes API, holds the connection
  while Postgres boots, pipes bytes, and scales back to zero after an idle window.

## The four scaling axes

- **Write** — vertical (in-place resize + autoscaler).
- **Read** — horizontal read-replica pool (`DATABASE_URL_RO`).
- **Tenant** — branch-per-app: N apps as Neon branches on one shared plane, each with
  its own 0↔1 compute, provisioned by the AppDatabase operator.
- **Zone** — eventual cross-zone consistency via logical replication (SCS zones).

See [Scaling](/docs/scale-zero-pg/scaling) for the full model.

## Bind it to a knext app

A knext `NextApp` declares its database inline; the operator provisions a
branch-per-app Postgres and injects `DATABASE_URL` (and `DATABASE_URL_RO`) into the
app. See [Connecting](/docs/scale-zero-pg/connecting) and the
[AppDatabase API](/docs/scale-zero-pg/appdatabase-api). To query it with a typed client —
schema, migrations, and read/write routing — use the [Data SDK](/docs/data-sdk).

## A typed data layer + extensions

Apps talk to the database through **[`@knext/db`](/docs/scale-zero-pg/data-sdk)** — a thin
[Drizzle](https://orm.drizzle.team) wrapper over the scale-to-zero pools: a typed schema,
writer-only `kn-next db migrate`, and an explicit `getDb()` (writer, read-your-writes) /
`getDbRO()` (bounded-stale reader) split. Two extensions ship in the compute image —
**[TimescaleDB and pgvector](/docs/scale-zero-pg/extensions)** — which an app self-enables
over its own `DATABASE_URL`, and which survive scale-to-zero.
