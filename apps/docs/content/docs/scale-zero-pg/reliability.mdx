Waking a database on a client connection is the one moment scale-to-zero has to get right.
This page covers the reliability machinery around that path — what absorbs transient
failures, what makes cold boots deterministic, and how the platform stays consistent under
churn. All of it is transparent to your app: a well-configured client (connect timeout
≥ 10s) never sees it.

## Wake-on-connect, in one paragraph

The gateway parses the Postgres startup packet, wakes the compute through the Kubernetes
API, holds the connection while Postgres boots, pipes bytes, and scales back to zero after
an idle window. A first connection to a sleeping database blocks ~2.5s (or ~0.4s from an
opt-in warm tier) and then completes normally — **no error, no client retry needed**. See
[Connecting](/docs/scale-zero-pg/connecting) for the app-facing contract.

## Bounded retry on the wake scale call

A transient apiserver blip on the `0→1` scale call — a TLS handshake timeout, a 5xx, a
throttle, a `Conflict`, a context deadline — used to fail the client's cold wake. The
gateway now wraps the scale call in a **bounded, idempotent retry** with exponential
backoff + jitter:

- The `GetScale → UpdateScale` sequence is idempotent, so retrying is safe.
- Total retry time is **clamped to `GW_WAKE_TIMEOUT_MS`** — a genuinely-down apiserver
  still fails **bounded** (no hang).
- A **terminal** error (`NotFound` / `Forbidden` / `Unauthorized` / …) fails loud
  immediately, with no retry.
- A healthy wake is unchanged — retry fires only on error.

The `pggw_wake_retries_total` metric counts blips absorbed. Rising while
`pggw_wake_failures_total` stays flat means retries are silently rescuing wakes the old
path would have failed; rising *together* means a sustained apiserver outage where retries
are exhausting.

## Deterministic cold-boot readiness gate (opt-in)

On a genuine cold wake there is a brief window while `compute_ctl` applies the per-app role
before the first auth attempt. By default the gateway absorbs this with a fixed 250 ms
settle (`GW_ROLE_APPLY_SETTLE_MS`), which makes the race negligible but not provably zero.

An **opt-in deterministic gate** replaces the blind sleep: when configured, the gateway
polls `compute_ctl`'s `/status` endpoint (port 3080) until it reports `running` — i.e. the
spec/role apply is provably done — and proceeds the instant it is, bounded by
`GW_WAKE_TIMEOUT_MS`.

<Callout type="info">
**Belt-and-suspenders.** The gate is deterministic when `/status` answers, and falls back
to the bounded 250 ms settle if `/status` is unreachable or rejects its token — it can
never wedge a wake. It is **off by default** (`GW_STATUS_PORT` unset; enabling it also
means exposing 3080 on the compute Service + NetworkPolicy and wiring the JWT), so the
shipped cold-wake numbers stand unchanged.
</Callout>

## Wake-budget side-channel guard

Because a wake is externally triggerable, an unauthenticated client on the open front door
could otherwise pay to spin compute up. A per-app **wake budget** (`GW_WAKE_BUDGET`,
deployed at `15` on the apps-gateway) caps the burst of `0→1` wakes an app may trigger,
refilling over `GW_WAKE_WINDOW_MS`. Excess wakes are refused `53400` **without scaling** —
so a flood can't turn the wake path into a cost or resource lever. This pairs with the
[no-tenant-existence-oracle](/docs/scale-zero-pg/connecting) guarantees on the auth path.

## Drift verification — "exists" vs "healthy"

Manifests existing is not the same as the plane being healthy. `_verify-drift.sh` asserts
the deployment is pointed at the live plane and checks child health, and a `--deep` mode
wakes a bounded sample of app computes to confirm they actually serve. Drift found in
verification is treated as a defect, not a warning.

## Cascade-safe teardown (operator ownerReferences)

The AppDatabase operator stamps a **controller `ownerReference`** on every child it creates
for an app — the per-app Secret, ConfigMap, writer Deployment + Service, and (when the RO
pool is enabled) the RO Deployment / Service / HPA. So Kubernetes-native **cascade
garbage-collection** reaps all children when the `AppDatabase` CR is deleted — defense in
depth over the finalizer path (which could otherwise orphan children if the finalizer is
force-removed while the operator is down). Owner refs are set on both create and update, so
apps provisioned earlier (or via the break-glass `provision-app.sh`) converge on the next
reconcile.
