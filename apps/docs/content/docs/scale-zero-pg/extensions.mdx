The scale-to-zero Postgres ships two extensions in its compute image — **TimescaleDB**
(time-series) and **pgvector** (embeddings / semantic search). Both are **opt-in and
self-service**: your app enables the one it needs **itself**, once, over its own
`DATABASE_URL` — no operator, no superuser. Both are `trusted`, so a single
`CREATE EXTENSION IF NOT EXISTS …` is all it takes.

<Callout type="info">
**They survive scale-to-zero.** Extension objects, data, and indexes live on the
pageserver (per-branch catalog + tables), so scaling the compute to zero is a restart, not
data loss. After a wake, hypertables, `time_bucket()`, vector indexes, and `pg_extension`
rows are all intact — the same durability the base wake drill proves. Verified live on OKE
(scale-zero-pg v1.4.0, `deploy/_verify-extensions.sh`).
</Callout>

## Self-enable — no superuser

An app enables an extension over its own per-app credential. The branch template grants
`CREATE` on the app database, so the per-app role can run `CREATE EXTENSION` without
`cloud_admin`:

```sql
-- run once, over your own DATABASE_URL (writer)
CREATE EXTENSION IF NOT EXISTS timescaledb;
CREATE EXTENSION IF NOT EXISTS vector;
```

pgvector needs **no preload and no image bump** — `CREATE EXTENSION vector` works with the
stock compute config (it is not in `shared_preload_libraries`).

## From `@knext/db`

If you use the [data SDK](/docs/scale-zero-pg/data-sdk), `@knext/db/schema` emits the
enable + index SQL for you. These are **migration SQL emitters** (drizzle-kit can't model
them) — put their output at the top of the migration that creates the table.

### TimescaleDB

```ts
import {
  pgTable, timestamp, text, doublePrecision,
  hypertable, dropChunks, createTimescaleExtension,
} from '@knext/db/schema';

export const metrics = pgTable('metrics', {
  ts: timestamp('ts', { withTimezone: true }).notNull(),
  device: text('device').notNull(),
  value: doublePrecision('value').notNull(),
});

createTimescaleExtension();                          // CREATE EXTENSION IF NOT EXISTS timescaledb;
hypertable(metrics, { by: 'ts', chunkInterval: '7 days' });
dropChunks(metrics, { olderThan: '30 days' });       // one-shot retention — run from CI/a job
```

<Callout type="warn">
**Apache-2 tier only.** You get hypertables, `time_bucket()`, chunk pruning, and one-shot
`drop_chunks()` retention. Columnar **compression** and **continuous aggregates** are
**not** available, and retention is `dropChunks()` — **not** `add_retention_policy()`.
Those rely on *background policy jobs* (TimescaleDB TSL), which cannot run on a compute
that scales to zero. Run `dropChunks()` from a CI cron or a `kn-next` job. pgvector has no
such caveat (no background worker).
</Callout>

### pgvector

```ts
import {
  pgTable, serial, text, vector,
  hnsw, ivfflat, createVectorExtension,
} from '@knext/db/schema';

export const docs = pgTable('docs', {
  id: serial('id').primaryKey(),
  body: text('body').notNull(),
  embedding: vector('embedding', { dimensions: 1536 }),
});

createVectorExtension();                             // CREATE EXTENSION IF NOT EXISTS vector;
hnsw('docs_embedding_idx', docs.embedding, { m: 16, efConstruction: 64 });
// IVFFlat is the alternative access method (faster build; tune `lists` to row count):
ivfflat('docs_embedding_ivf', docs.embedding, { ops: 'vector_l2_ops', lists: 100 });
```

Query with the distance operators (re-exported from drizzle). **Match the operator to the
index's ops class**: `cosineDistance` (`<=>`) ⇄ `vector_cosine_ops`, `l2Distance` (`<->`)
⇄ `vector_l2_ops`, `innerProduct` (`<#>`) ⇄ `vector_ip_ops`.

```ts
import { getDbRO, cosineDistance } from '@knext/db';
import { docs } from '@/db/schema';

const nearest = await getDbRO({ docs })
  .select().from(docs)
  .orderBy(cosineDistance(docs.embedding, queryEmbedding))
  .limit(5);
```

Build the index while the compute is awake (index builds run on your own per-app compute);
it persists across scale-to-zero like any other table.

<Callout type="info">
**Version gate.** The self-enable trusted-extension mechanism landed in
**scale-zero-pg ≥ v1.4.0**. The `@knext/db` helpers are designed + unit-tested regardless;
the live `CREATE EXTENSION` needs the extension present on the compute image.
</Callout>
